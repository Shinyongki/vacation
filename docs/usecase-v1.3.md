### UC-01. 로그인

**액터**: 전체  
**선행 조건**: 시스템이 실행 중이고 같은 네트워크에서 브라우저 접속 가능  
**후행 조건**: 역할별 대시보드 표시  

**기본 흐름**:
1. 사용자가 브라우저로 시스템에 접속한다
2. 시스템이 로그인 화면(C-01)을 표시한다
3. 사용자가 사번과 비밀번호를 입력한다
4. 시스템이 employees 테이블에서 사번을 조회한다
5. 비밀번호 해시를 비교하여 인증한다
6. 시스템이 해당 직원의 role을 확인한다
7. 역할에 맞는 대시보드로 이동한다 (D-01~05)
8. 사이드바(C-02)에 역할별 메뉴를 표시한다

**대안 흐름**:
- 4a. 사번이 존재하지 않는 경우 → "사번 또는 비밀번호가 올바르지 않습니다" 표시
- 5a. 비밀번호 불일치 → 동일 에러 메시지 표시 (사번 존재 여부 노출 방지)
- 6a. status가 inactive인 경우 → "비활성화된 계정입니다. 관리자에게 문의하세요" 표시

---

### UC-04. 초기 비밀번호 로그인

**액터**: 전체 (신규 등록 직원)  
**선행 조건**: HR관리자가 직원을 등록한 직후  
**후행 조건**: 비밀번호 변경 완료 후 대시보드 진입  

**기본 흐름**:
1. 신규 직원이 사번과 초기 비밀번호(생년월일 6자리)로 로그인한다
2. 시스템이 초기 비밀번호 여부를 감지한다
3. 비밀번호 변경 모달(C-04)을 강제 표시한다
4. 직원이 새 비밀번호를 입력하고 확인한다
5. 변경 완료 후 역할별 대시보드로 이동한다

### UC-10. 종일 연차 신청

**액터**: 직원  
**선행 조건**: 로그인 완료, 잔여 연차 ≥ 신청 일수  
**후행 조건**: 결재 라인 1단계 결재자에게 알림 발송  

**기본 흐름**:
1. 직원이 대시보드(D-01) 또는 내 휴가 목록(L-01)에서 "휴가 신청" 버튼을 클릭한다
2. 시스템이 신청 폼 모달(L-03)을 표시한다
   - **직원의 성별에 따라 사용 불가 유형 비활성화 (남성: 생리휴가·출산휴가)**
   - **직원의 고용형태에 따라 HR관리자가 설정한 미허용 유형 비활성화**
3. 직원이 유형에서 "연차"를 선택한다
4. 하위 옵션에서 "종일"을 선택한다
5. 달력 위젯에서 시작일~종료일을 범위 선택한다
6. 시스템이 실시간으로 계산하여 표시한다:
   - 공휴일·자체 휴일 자동 제외한 차감 일수
   - 잔여 일수 프로그레스 바 (신청 전 → 신청 후)
   - 해당 기간 팀 출근 현황
7. 직원이 사유를 입력한다 (선택)
8. **직원이 열람 범위를 설정한다 (팀별 체크박스)**
   - 내 팀: 기본 체크 (해제 가능)
   - 다른 팀: 필요 시 체크
   - 안내: "승인 후 선택한 팀의 캘린더에 이름·유형·기간이 표시됩니다"
9. 직원이 "상신" 버튼을 클릭한다
10. 시스템이 leave_requests에 기록하고, **leave_request_visibility에 선택한 팀을 기록**하고, 해당 휴가 유형의 결재 라인에 따라 approval_steps를 생성한다
11. 1단계 결재자(기안 다음 단계)에게 알림을 발송한다
12. 내 휴가 목록(L-01)으로 이동하며 "승인 대기" 상태로 표시된다

**대안 흐름**:
- 6a. 잔여 일수가 부족한 경우 → 차감 일수가 빨간색으로 표시, 제출 버튼 비활성화
- 6b. 동시 휴가 인원 제한이 설정되어 있고 초과하는 경우 → 경고 메시지 표시 (제출은 가능)
- 9a. 선택한 날짜가 과거인 경우 → UC-17 사후 휴가 신청으로 분기

### UC-11. 반차(오전/오후) 신청

**액터**: 직원  
**선행 조건**: 로그인 완료, 잔여 연차 ≥ 0.5일  
**후행 조건**: 결재 라인 1단계 결재자에게 알림 발송  

**기본 흐름**:
1~3. UC-10과 동일
4. 하위 옵션에서 "오전반차" 또는 "오후반차"를 선택한다
5. 달력 위젯에서 단일 날짜를 선택한다
6. 시스템이 표시한다:
   - 오전반차: 09:00~14:00 / 오후반차: 14:00~18:00 (설정된 경계 시각 기준)
   - 차감 일수: 0.5일
   - 잔여 일수 프로그레스 바
7~12. UC-10과 동일 (열람 범위 설정 포함)

### UC-12. 시간연차 신청

**액터**: 직원  
**선행 조건**: 로그인 완료, 잔여 연차 > 0  
**후행 조건**: 결재 라인 1단계 결재자에게 알림 발송  

**기본 흐름**:
1~3. UC-10과 동일
4. 하위 옵션에서 "시간연차"를 선택한다
5. 달력 위젯에서 단일 날짜를 선택한다
6. 시작 시각과 종료 시각을 30분 단위 드롭다운으로 선택한다
7. 시스템이 계산한다:
   - 점심시간(12:00~13:00)이 포함된 경우 자동 제외
   - 실제 사용 시간 계산
   - 차감 일수 계산 (사용 시간 ÷ 8시간) 실시간 표시
8~12. UC-10과 동일 (열람 범위 설정 포함)

**대안 흐름**:
- 7a. 누적 시간연차 합산 결과가 8시간 이상이 되는 경우 → "8시간 도달 시 연차 1일 차감" 안내 표시

### UC-13. 경조사 휴가 신청

**액터**: 직원  
**선행 조건**: 로그인 완료  
**후행 조건**: 경조사 결재 라인의 1단계 결재자에게 알림 발송  

**기본 흐름**:
1~2. UC-10과 동일
3. 직원이 유형에서 "경조사"를 선택한다
4. 세부 유형 드롭다운이 표시된다 (condolence_subtypes 기반)
5. 세부 유형을 선택하면 규정 일수가 녹색 박스로 즉시 표시된다
6. 달력 위젯에서 시작일을 선택하면 종료일이 규정 일수에 맞춰 자동 설정된다
7. 사유를 입력한다 (필수)
8. 확인서류 첨부 안내가 표시된다
9~12. UC-10과 동일 (열람 범위 설정 포함, 경조사 결재 라인 적용)

**대안 흐름**:
- 6a. 배우자 출산의 경우 → "3회까지 분할 사용 가능" 안내 표시, 분할 날짜 선택 가능

### UC-14. 병가 신청

**액터**: 직원  
**선행 조건**: 로그인 완료  
**후행 조건**: 병가 결재 라인의 1단계 결재자에게 알림 발송  

**기본 흐름**:
1~2. UC-10과 동일
3. 직원이 유형에서 "병가"를 선택한다
4. 달력 위젯에서 기간을 선택한다
5. 사유를 입력한다 (필수)
6. 시스템이 누적 병가 일수를 계산하여 표시한다
7~12. UC-10과 동일 (열람 범위 설정 포함, 병가 결재 라인 적용)

**대안 흐름**:
- 6a. 연간 누적 6일 초과 시 → "진단서 첨부가 필요합니다" 필수 안내 + 첨부 영역 강조
- 6b. 연간 60일 초과 시 → "일반 병가 한도(60일)를 초과합니다" 경고
- 6c. 업무상 부상·질병인 경우 → 별도 체크 → 180일 한도 적용

### UC-16. 긴급 휴가 신청

**액터**: 직원  
**선행 조건**: 로그인 완료, 휴가 시작일이 임박  
**후행 조건**: 결재자 + 대결자에게 동시 알림. 부재 결재자는 즉시 대결자 전환  

**기본 흐름**:
1~6. UC-10/11/12와 동일 (유형에 따라)
7. 직원이 "긴급" 체크박스를 체크한다
8. 긴급 사유 입력란이 표시된다 (필수)
9. 긴급 사유를 입력한다
10. 열람 범위를 설정한다
11. "상신" 버튼을 클릭한다
12. 시스템이 결재자와 대결자에게 동시에 "긴급" 알림을 발송한다
13. 결재자가 부재중 상태인 경우 대기 없이 즉시 대결자에게 전환한다
14. 결재 목록에서 해당 건에 "긴급" 배지가 표시된다

### UC-17. 사후 휴가 신청

**액터**: 직원  
**선행 조건**: 로그인 완료, 이미 지난 날짜에 대한 휴가 신청  
**후행 조건**: "사후 신청" 배지 부착된 결재 건 생성  

**기본 흐름**:
1~2. UC-10과 동일
3. 직원이 유형을 선택한다
4. 달력 위젯에서 과거 날짜를 선택한다
5. 시스템이 사후 신청을 자동 감지하고 안내 메시지를 표시한다
6. 사후 신청 사유 카테고리 6가지가 표시된다
7. 직원이 해당 카테고리를 선택한다 (필수)
8. 상세 사유를 입력한다 (필수)
9. 증빙 서류 첨부 안내가 표시된다
10. 열람 범위를 설정한다
11. "상신" 버튼을 클릭한다
12. 결재 목록에서 해당 건에 "사후 신청" 배지가 표시된다

**대안 흐름**:
- 3a. 선택한 유형이 사후 신청 불가 유형인 경우 (연차, 포상휴가, 생리휴가) → "해당 유형은 사후 신청이 불가합니다" 표시, 제출 버튼 비활성화

### UC-18. 휴가 취소

**액터**: 직원  
**선행 조건**: 본인이 신청한 휴가 건이 존재  
**후행 조건**: 해당 건 상태가 "취소"로 변경, 잔여일수 복원  

**기본 흐름**:
1. 직원이 내 휴가 목록(L-01)에서 취소할 건을 선택한다
2. 신청 상세(L-02)에서 "취소" 버튼을 클릭한다
3. 취소 확인 모달(L-04)이 표시된다
4. 취소 사유를 입력한다
5. "확인" 버튼을 클릭한다
6. 시스템이 해당 건 상태를 "cancelled"로 변경한다
7. 차감되었던 잔여일수를 복원한다
8. 관련 결재자에게 취소 알림을 발송한다

**대안 흐름**:
- 1a. 이미 최종 승인된 건의 경우 → 취소 버튼 대신 "취소 재신청" 안내 표시
- **1b. 수정 목적인 경우 → "회수" 버튼 사용 안내 (UC-18a)**

**※ 취소 vs 회수 구분**: 취소(cancelled)는 건이 완전히 종료되어 재기안 불가. 수정해서 다시 올리려면 회수(recalled)를 사용.

### UC-18a. 휴가 회수 (v1.1 신규)

**액터**: 직원·팀장·원장·HR (기안자 본인만)  
**선행 조건**: 본인이 신청한 건이 pending 상태 (결재 진행 중, 최종 승인 전)  
**후행 조건**: 건 상태가 recalled로 변경, 결재 단계 초기화, 결재자에게 알림  

**기본 흐름**:
1. 기안자가 내 휴가 목록(L-01)에서 pending 건을 선택한다
2. 신청 상세(L-02)에서 "회수" 버튼을 클릭한다
3. 회수 확인 모달(L-05)이 표시된다
   - 안내: "이 건을 회수하면 결재가 초기화됩니다. 회수 후 '재기안'으로 수정하여 다시 올릴 수 있습니다."
4. 회수 사유를 입력한다 (선택)
5. "확인" 버튼을 클릭한다
6. 시스템이 해당 건 상태를 "recalled"로 변경한다
7. 진행 중이던 approval_steps를 모두 초기화한다
8. 관련 결재자에게 "회수됨" 알림을 발송한다
9. 내 휴가 목록에서 해당 건에 "재기안" 버튼이 표시된다

**대안 흐름**:
- 2a. 이미 최종 승인된 건 → "회수" 버튼 비활성화, "이미 승인된 건은 회수할 수 없습니다" 안내
- 2b. 이미 반려된 건 → "회수" 버튼 대신 "재상신" 버튼 표시

### UC-18b. 재기안 — 회수 건 (v1.1 신규)

**액터**: 직원·팀장·원장·HR (기안자 본인만)  
**선행 조건**: 본인이 회수한 건 (recalled 상태)  
**후행 조건**: 이전 데이터가 복사된 새 건 생성 (pending), 원본과 parent_request_id로 연결  

**기본 흐름**:
1. 기안자가 내 휴가 목록(L-01)에서 recalled 건의 "재기안" 버튼을 클릭한다
2. 신청 폼 모달(L-03)이 열리며, 원본의 데이터가 미리 채워진다:
   - 유형, 날짜/시간, 사유, 열람 범위 등 모든 필드
3. 기안자가 필요한 부분을 수정한다 (예: 날짜 변경)
4. "상신" 버튼을 클릭한다
5. 시스템이 새 leave_requests 레코드를 생성한다:
   - status: pending
   - parent_request_id: 원본 건 ID
6. leave_request_visibility에 열람 범위를 기록한다
7. 해당 유형의 결재 라인에 따라 새 approval_steps를 생성한다
8. 1단계 결재자에게 알림을 발송한다
9. 내 휴가 목록에서 새 건이 "승인 대기" 상태로 표시된다
10. 원본 recalled 건은 유지되며, 새 건에서 "원본 보기" 링크로 접근 가능

### UC-18c. 재상신 — 반려 건 (v1.1 신규)

**액터**: 직원·팀장·원장·HR (기안자 본인만)  
**선행 조건**: 본인이 신청하여 반려된 건 (rejected 상태)  
**후행 조건**: 이전 데이터가 복사된 새 건 생성 (pending), 원본과 parent_request_id로 연결  

**기본 흐름**:
1. 기안자가 내 휴가 목록(L-01)에서 rejected 건의 "재상신" 버튼을 클릭한다
2. 신청 폼 모달(L-03)이 열리며:
   - 상단에 반려 사유가 노란 안내 박스로 표시된다
   - 원본의 데이터가 미리 채워진다 (유형, 날짜/시간, 사유, 열람 범위)
3. 기안자가 반려 사유를 참고하여 수정한다 (예: 날짜 변경)
4~10. UC-18b와 동일

### UC-19. 신청 상태 추적

**액터**: 직원  
**선행 조건**: 본인이 신청한 건이 존재  
**후행 조건**: 없음 (조회만)  

**기본 흐름**:
1. 직원이 내 휴가 목록(L-01)에서 건을 선택한다
2. 신청 상세(L-02)가 **좌우 2단 레이아웃**으로 표시된다
3. **좌측: 결재 현황 타임라인** — 결재 라인 진행 상태가 단계별로 표시된다:
   - 각 단계: 결재 유형 (기안/협조/검토/결재) + 결재자 이름·직위
   - 상태: 완료(체크) / 진행중(강조) / 대기(회색)
   - 열람 시각 + 결재 시각 분리 표시
   - 대결 처리 건: "원래 담당자 (대결: 실제 처리자)" 표시
4. **우측: 신청 정보** — info-table 구조로 3개 섹션 표시:
   - 신청자 정보 (이름, 소속, 신청일)
   - 휴가 정보 (유형, 기간, 신청 일수, 사유)
   - 추가 정보 (긴급 사유, 사후 신청 사유, 회수 사유 — 해당 시만)
5. parent_request_id가 있으면 "원본 보기" 링크가 표시된다
6. pending 상태이면 "회수" 버튼, rejected이면 "재상신" 버튼, recalled이면 "재기안" 버튼이 표시된다 (기안자 본인만)
7. **"목록으로" 버튼은 navigate(-1) (브라우저 뒤로가기) — 진입 경로에 따라 자동 복귀**

### UC-20. 개별 승인

**액터**: 팀장·원장  
**선행 조건**: 본인에게 할당된 결재 대기 건이 존재  
**후행 조건**: 다음 단계 결재자에게 알림 발송 (또는 최종 승인 완료)  

**기본 흐름**:
1. 팀장/원장이 알림을 클릭하거나 승인 대기 목록(A-01)에 진입한다
2. 대기 건 목록이 표시된다 (신청자, 유형, 기간, 대기일수, 긴급/사후 배지)
3. 건을 선택하여 신청 상세(L-02)를 확인한다
4. "승인" 버튼을 클릭한다
5. 시스템이 approval_steps 상태를 "approved"로 변경하고 acted_at을 기록한다
6. 결재 당시 직위·부서를 스냅샷으로 저장한다
7. 다음 단계가 있으면 다음 결재자에게 알림을 발송한다
8. 마지막 단계이면 leave_requests 상태를 "approved"로 변경하고, 잔여일수를 차감한다
9. 신청자에게 "승인 완료" 알림을 발송한다

8. 해당 사번으로 즉시 로그인이 가능해진다

**대안 흐름**:
- 4a. 사번 중복 시 → "이미 등록된 사번입니다" 표시

### UC-51. 보고서 내보내기 (엑셀+PDF)

**액터**: 원장·HR관리자·재단담당자  
**선행 조건**: 보고서 메뉴 접근 권한  
**후행 조건**: zip 파일 다운로드 + export_logs에 이력 기록  

**기본 흐름**:
1. 사용자가 보고서 메뉴(R-01 또는 R-02)에 진입한다
2. 기간과 유형 (사용 현황 / 요약)을 선택한다
3. 보고서 데이터를 확인한다
4. "내보내기" 버튼을 클릭한다
5. 시스템이 동시에 2개 파일을 생성한다 (각각 5개 섹션/시트 구조):
   - 엑셀 (.xlsx): 5시트 — 총괄요약, 사용상세, 결재이력, 조정이력, 검증정보
   - PDF (.pdf): 5섹션 — 동일 구조
6. **섹션 1 — 표지+총괄요약**: 시설명, 기간, 생성자/일시, 직원별 발생일수·사용일수·조정일수·잔여일수·소진율
7. **섹션 2 — 사용 상세**: 건별 내역 + 결재완료일·최종승인자 (대결/사후/긴급 표시)
8. **섹션 3 — 결재 이력 상세**: 건별 결재 단계 추적 (열람시각·결재시각, 대결 여부, 즉시처리 표시) + 대결 비율/즉시처리 비율 통계
9. **섹션 4 — 수동 조정 이력**: 조정일시·대상·조정자·조정량·사유 + 자동 경고 (대량 조정, 반복 조정, 사유 불명확)
10. **섹션 5 — 검증 정보**: 생성일시, 생성자, 데이터 기간, 데이터 요약(총 조정 건수 포함), 검증코드(SHA-256), 검증 방법 안내
11. **검증코드는 총괄요약+사용상세+결재이력+조정이력 4개 데이터셋을 JSON.stringify → HMAC-SHA256**
12. export_logs에 이력을 자동 기록한다
13. 2개 파일이 zip으로 묶여 다운로드된다

### UC-60. 연차 자동 계산

**액터**: 시스템  
**트리거**: 연도 시작 시 (매년 1월 1일), 직원 등록 시, 관리자 재계산 요청 시  
**후행 조건**: leave_balances 갱신  

**기본 흐름**:
1. 시스템이 employees 테이블에서 active 상태 직원 목록을 조회한다
2. 각 직원의 입사일 기준 근속연수를 계산한다
3. 규정에 따라 올해 발생 일수를 계산한다:
   - 1년 미만: 월 1일 발생
   - 1년 이상: 15일
   - 3년 이상: 2년마다 +1일
   - 최대 25일
4. leave_balances의 total_days를 갱신한다
5. calc_detail에 계산 과정을 JSON으로 기록한다

### UC-62. 미처리 자동 대결 전환

**액터**: 시스템  
**트리거**: 주기적 확인 (예: 매 30분)  
**후행 조건**: 미처리 건이 대결자에게 전환  

**기본 흐름**:
1. 시스템이 approval_steps에서 status가 "pending"인 건을 조회한다
2. 각 건의 대기 시간을 계산한다
3. 해당 건의 leave_requests에서 휴가 시작일까지 남은 일수를 확인한다
4. 동적 전환 시간을 적용한다:
   - 휴가 시작 1일 이내: 즉시 전환
   - 휴가 시작 2~3일: 8시간 경과 시 전환
   - 휴가 시작 4일 이상: 24시간 경과 시 전환
5. 전환 시간의 50% 경과 시 원 결재자에게 "미처리 경고" 알림을 발송한다
6. 전환 시간 도달 시:
   - 대결자(1순위)에게 결재 건을 전환하고 알림을 발송한다
   - 1순위 대결자도 부재 시 2순위로 전환한다
   - 대결자도 모두 부재 시 원 결재자에게 대기 유지

---

### UC-63. 부재 복귀 예정일 자동 해제

**액터**: 시스템  
**트리거**: 매일 자정 (또는 주기적 확인)  
**후행 조건**: 부재 상태 자동 해제  

**기본 흐름**:
1. 시스템이 employees에서 is_absent가 true인 직원을 조회한다
2. absent_return_date가 오늘 이전인 직원을 필터한다
3. 해당 직원의 is_absent를 false로, absent_return_date를 null로 변경한다
4. 부재 해제 시점에 대기 중인 결재 건이 있으면 알림을 발송한다
