# 데이터 무결성 검증 가이드

> 재단법인 경상남도사회서비스원 — 재단담당자용  
> 작성일: 2026.04.11  
> 버전: 1.0  
> 관련 문서: PRD v1.0, IA v1.0

---

## 1. 개요

### 1.1 이 문서의 목적

휴가관리 시스템에서 시설이 제출하는 보고서 데이터가 정확한지, 중간에 조작되지 않았는지를 재단담당자가 확인할 수 있는 방법을 안내한다.

### 1.2 왜 필요한가

시설의 HR관리자는 직원 등록, 잔여일수 조정, 보고서 내보내기 등 광범위한 권한을 갖고 있다. 보고서를 엑셀로 내보낸 뒤 수정하여 제출하거나, DB 데이터 자체를 사전에 조작할 가능성을 원천적으로 차단할 수는 없지만, **"조작하면 흔적이 남고, 검증하면 들킨다"**는 억제 구조를 통해 데이터 신뢰성을 확보한다.

### 1.3 검증 체계 구조

```
┌─────────────────────────────────────────────────────────┐
│                    3중 검증 구조                          │
├──────────────┬──────────────┬───────────────────────────┤
│  1. 검증코드  │  2. 이력 추적  │  3. 직접 확인              │
│  (원본 증빙)  │  (행위 추적)   │  (현장 확인)              │
├──────────────┼──────────────┼───────────────────────────┤
│ PDF에 삽입된  │ 내보내기 이력  │ 재단담당자가 직접         │
│ 해시 검증코드 │ + 조정 이력   │ 시스템 로그인하여         │
│ 로 원본 확인  │ 로 이상 감지  │ 보고서 직접 내보내기      │
└──────────────┴──────────────┴───────────────────────────┘
```

## 2. 보고서 수령 시 확인사항

### 2.1 시설에서 받는 파일 형태

시설이 보고서를 내보내면 시스템이 **2개 파일을 동시에 생성**한다.

| 파일 | 용도 | 특징 |
|------|------|------|
| 엑셀 (.xlsx) | 재단의 취합·분석 작업용 | 수정 가능 — 공식 증빙 아님 |
| PDF (.pdf) | 원본 증빙용 | 검증코드 삽입됨 — 공식 증빙 |

두 파일은 zip으로 묶여서 다운로드되며, **동일한 쿼리 결과**로 생성되므로 내용이 정확히 일치해야 한다.

### 2.2 수령 즉시 확인할 것

1. **파일이 2개인가?** — 엑셀만 왔다면 시스템에서 내보낸 것이 아닐 수 있음
2. **PDF 마지막 페이지에 검증 정보가 있는가?** — 아래 항목이 모두 표시되어야 함:
   - 생성일시 (예: 2026-04-11 14:30:22)
   - 생성자 (사번 + 이름)
   - 데이터 기간 (예: 2026-01-01 ~ 2026-03-31)
   - 데이터 요약 (총 직원 수, 총 사용일수 합계)
   - 검증코드 (예: `A3F8-B2C1-D9E4-7F1A`)
3. **엑셀과 PDF의 합계가 일치하는가?** — 총 직원 수, 총 사용일수 등 핵심 수치 대조

## 3. 검증 방법 상세

### 3.1 방법 1 — 검증코드로 원본 확인 (원격 불가, 현장 방문 시)

**목적**: 받은 PDF가 시스템에서 생성된 원본인지 확인

**절차**:
1. 시설 방문 → 시설 네트워크에 접속
2. 브라우저에서 시스템 접속 → 재단담당자 계정으로 로그인
3. 사이드바 "데이터 검증" 메뉴 클릭
4. "보고서 검증" 탭 진입
5. 받았던 PDF 하단의 검증코드를 입력
6. "검증" 버튼 클릭

**결과 해석**:

| 결과 | 의미 | 조치 |
|------|------|------|
| 원본 확인 (일치) | 이 검증코드와 동일한 데이터가 시스템에서 생성된 기록이 있음 | 정상 |
| 데이터 변경 감지 (불일치) | 이 검증코드에 해당하는 기록이 없거나, 데이터가 변경된 후 재생성됨 | 시설에 소명 요청 |
| 검증코드 없음 | 입력한 코드 형식이 올바르지 않음 | PDF가 시스템 생성 문서인지 재확인 |

**원리 설명**:
- 시스템이 보고서를 생성할 때, 모든 데이터를 시스템 비밀키와 조합하여 해시(SHA-256)를 계산한다
- 이 해시값이 검증코드로 PDF에 삽입되고, 동시에 export_logs 테이블에 저장된다
- 검증 시 입력된 코드와 export_logs의 기록을 대조한다
- 비밀키는 .exe 실행파일 내부에만 존재하며 화면이나 DB에 노출되지 않는다
- 데이터가 1글자만 달라져도 해시 결과가 완전히 달라지므로, 수정 후 동일한 검증코드를 만들어내는 것은 불가능하다

### 3.2 방법 2 — 내보내기 이력 확인 (원격 불가, 현장 방문 시)

**목적**: 보고서가 비정상적으로 여러 번 출력되었는지 확인

**절차**:
1. 시스템 로그인 → "데이터 검증" → "내보내기 이력" 탭
2. 기간 설정 (예: 최근 3개월)
3. 내보내기 기록 목록 확인

**확인 포인트**:

| 상황 | 의미 | 조치 |
|------|------|------|
| 같은 기간 데이터를 1회만 내보냄 | 정상 | — |
| 같은 기간 데이터를 여러 번 내보냄 | 처음 내보낸 후 데이터를 수정하고 재출력했을 가능성 | 시설에 사유 확인 |
| 제출받은 보고서의 생성일시와 이력의 일시가 다름 | 다른 시점에 생성된 보고서가 제출됨 | 어떤 버전이 최종인지 확인 |

**이력에 기록되는 정보**:
- 내보낸 사람 (사번 + 이름)
- 내보낸 일시
- 데이터 기간 (시작일 ~ 종료일)
- 보고서 유형 (사용 현황 / 요약)
- 검증코드

**시스템 자동 경고**: 동일 기간 데이터가 2회 이상 내보내진 경우 해당 행에 경고 표시

### 3.3 방법 3 — 조정 이력 확인 (원격 불가, 현장 방문 시)

**목적**: HR관리자가 보고 전에 데이터를 사전 조작했는지 확인

**배경**: HR관리자는 업무상 잔여일수를 수동으로 조정할 수 있다 (전년도 이월, 특별 부여 등). 이 자체는 정당한 행위이지만, 보고서 제출 직전에 대량 조정이 있었다면 의심 사유가 된다.

**절차**:
1. 시스템 로그인 → "데이터 검증" → "조정 이력" 탭
2. 기간 설정 (보고 대상 기간 또는 보고서 제출 직전 기간)
3. 조정 내역 목록 확인

**확인 포인트**:

| 상황 | 의미 | 조치 |
|------|------|------|
| 조정 건수가 적고 사유가 구체적 | 정상적인 업무 조정 | — |
| 보고서 내보내기 직전에 대량 조정 | 보고 수치를 맞추기 위한 조작 가능성 | 각 건의 사유 상세 확인 |
| 사유가 "기타", "조정" 등 불명확 | 정당한 사유 없이 조정했을 가능성 | 시설에 소명 요청 |
| 특정 직원에게 조정이 집중 | 특정인의 연차를 인위적으로 변경 | 해당 직원의 신청 이력과 대조 |

**이력에 기록되는 정보**:
- 조정 대상 직원 (사번 + 이름)
- 조정자 (HR관리자 사번 + 이름)
- 조정 일시
- 조정량 (예: +2일, -1일)
- 사유 (필수 입력)

**시스템 자동 경고**: 보고서 내보내기 기록 기준 7일 이내에 5건 이상 조정이 있었을 경우 경고 표시

**핵심**: balance_adjustments 테이블은 INSERT만 가능하고 UPDATE/DELETE가 불가하므로, 한번 기록된 조정 이력은 HR관리자라도 삭제할 수 없다.

### 3.4 방법 4 — 직접 내보내기 (가장 확실, 현장 방문 시)

**목적**: 시설에서 받은 엑셀과 시스템 원본 데이터를 직접 비교

**절차**:
1. 시스템 로그인 → "보고서" 메뉴
2. 동일한 기간·유형의 보고서를 직접 내보내기
3. 시스템이 생성한 엑셀과 시설에서 받았던 엑셀을 비교

**이 방법이 가장 확실한 이유**: 재단담당자가 직접 내보낸 데이터는 HR관리자가 개입할 여지가 없다. 동일 조건으로 뽑은 데이터가 다르다면 시설에서 받은 엑셀이 수정된 것이다.

## 4. 검증 시나리오별 대응

### 시나리오 A — 엑셀을 수정하여 제출한 경우

```
발생: HR관리자가 엑셀을 내보낸 후 일부 수치를 수정하여 재단에 제출
감지: 엑셀과 PDF의 합계 불일치 (수령 즉시 확인 가능)
확인: 현장 방문 시 직접 내보내기로 원본 대조
```

### 시나리오 B — 엑셀과 PDF 모두 수정하여 제출한 경우

```
발생: HR관리자가 엑셀을 수정하고, PDF도 별도로 만들어서 제출
감지: PDF에 검증코드가 없거나 형식이 다름 (수령 즉시 확인 가능)
확인: 현장 방문 시 검증코드 검증 → 불일치
```

### 시나리오 C — DB 데이터를 사전 조작 후 정상 내보내기한 경우

```
발생: HR관리자가 잔여일수를 수동 조정한 후 시스템에서 정상적으로 내보내기
감지: 검증코드는 유효함 (이 방법만으로는 감지 불가)
확인: 현장 방문 시 조정 이력 조회 → 보고 직전 대량 조정 확인
      각 조정 건의 사유가 타당한지 개별 확인
```

### 시나리오 D — 같은 기간 보고서를 여러 번 내보낸 경우

```
발생: 처음 내보낸 후 데이터 변경, 다시 내보내기하여 제출
감지: 현장 방문 시 내보내기 이력 조회 → 동일 기간 복수 출력 확인
확인: 각 내보내기의 검증코드가 다름 → 중간에 데이터가 변경되었음을 의미
      어떤 버전이 제출되었는지, 왜 다시 내보냈는지 소명 요청
```

## 5. 재단담당자 전용 화면 구성

### 5.1 사이드바 메뉴 (3개)

```
재단담당자 사이드바
  ├─ 시설 현황 (열람 전용)
  ├─ 보고서 (열람 + 내보내기)
  └─ 데이터 검증
       ├─ 보고서 검증 (X-01)
       ├─ 내보내기 이력 (X-02)
       └─ 조정 이력 조회 (X-03)
```

### 5.2 각 화면 정보 요소

**보고서 검증 (X-01)**
- 검증코드 입력 필드
- "검증" 버튼
- 결과 표시 영역: 원본 확인 (녹색) / 데이터 변경 감지 (빨간색)
- 일치 시: 해당 내보내기의 생성자·생성일시·데이터 기간 표시

**내보내기 이력 (X-02)**
- 기간 필터 (시작일~종료일)
- 이력 테이블: 내보낸 사람, 일시, 데이터 기간, 보고서 유형, 검증코드
- 동일 기간 반복 출력 행에 경고 배지
- 정렬: 최신순 기본

**조정 이력 조회 (X-03)**
- 기간 필터 (시작일~종료일)
- 이력 테이블: 대상 직원, 조정자, 일시, 조정량, 사유
- 보고 직전 대량 조정 경고 표시
- 직원별·조정자별 필터

## 6. 검증 체계의 한계와 보완

### 6.1 이 시스템이 막을 수 있는 것

- 엑셀 파일 수정 후 제출 → PDF 대조로 즉시 감지
- PDF 위조 → 검증코드 불일치로 감지
- 잔여일수 임의 조정 → 조정 이력에 모든 흔적 남음
- 보고서 반복 출력 (수정 후 재출력) → 내보내기 이력에 기록

### 6.2 이 시스템이 막을 수 없는 것

- HR관리자가 SQLite DB 파일을 직접 열어서 balance_adjustments나 export_logs 레코드를 삭제하는 경우 → DB 파일 자체에 대한 접근 제어는 OS 수준의 문제
- 정당한 사유로 위장한 소규모 조정 → 사유의 진위는 시스템이 판단할 수 없음

### 6.3 추후 보완 가능한 방안

- **클라우드 동기화**: 추후 인터넷 연결 시 재단이 원격으로 시스템에 접속하여 실시간 검증 가능
- **DB 파일 해시 주기적 백업**: DB 파일 자체의 해시를 별도 저장하여 DB 직접 수정 감지
- **감사 로그 분리 저장**: 핵심 이력 테이블을 별도 파일로 분리하여 메인 DB와 독립 보관
