# 휴가관리 시스템 — PRD (제품 요구사항 문서)

> 재단법인 경상남도사회서비스원 산하 시설 납품용  
> 작성일: 2026.04.11  
> 버전: 1.3 (2026.04.13 — 보고서 5섹션 구조, 알림 개선, 시스템설정 한글화, UI 레이아웃 변경)  
> 관련 문서: IA v1.3, Use Case v1.3, 데이터 무결성 검증 가이드 v1.0, 디자인 가이드 v1.2

## 1. 제품 개요

### 1.1 배경

재단법인 경상남도사회서비스원 산하 시설에서는 현재 휴가 관리를 엑셀 및 수기로 처리하고 있다. 결재 과정이 종이 기반이며, 연차 계산·잔여일수 추적·현황 파악에 시간이 과도하게 소요된다. 이를 전자결재 기반 휴가관리 시스템으로 전환하여 업무 효율을 높이고 결재 이력의 투명성을 확보한다.

### 1.2 제품 목표

- 휴가 신청~승인~기록 전 과정을 전자화
- 연차 발생·차감·잔여 일수를 자동 계산하여 투명하게 제공
- 결재 이력을 완전히 추적 가능한 감사 추적(Audit Trail) 구현
- 인터넷 없이 시설 내부 네트워크만으로 운영 가능한 독립 실행형 시스템
- 기존 엑셀 데이터를 이전할 수 있는 마이그레이션 기능 제공

### 1.3 제품 범위

**포함**
- 휴가 신청·승인·반려·회수·취소 전자결재
- 5개 역할 기반 접근 제어 (직원, 팀장, 원장, HR관리자, 재단담당자)
- 규정 기반 연차 자동 계산 + HR 수동 조정
- 앱 내 알림 (이메일 없음)
- 보고서 생성 및 엑셀 내보내기
- 기존 엑셀 데이터 업로드·이전
- 기안자 열람 범위 설정 (승인된 휴가의 팀 캘린더 공개 범위)
- 회수·재기안·재상신 기능

**제외 (추후 검토)**
- 클라우드 동기화 (재단 담당자 원격 접속)
- 이메일/SMS 알림
- 모바일 앱 (반응형 웹으로 대응)
- 근태관리 (출퇴근 기록)
- 급여 연동
- 임시보관(draft) 기능 — DB 스키마에 예약만, 구현 보류

## 2. 사용자 역할

| 역할 | 대상 | 핵심 니즈 | 첫 화면 |
|------|------|----------|---------|
| 직원 | 일반 직원 | 내 연차가 며칠 남았는지 정확히 알고 싶다 | 잔여 연차 (크게) + 계산 근거 |
| 팀장 | 팀 관리자 | 오늘 팀에 누가 나오는지, 승인할 건이 있는지 빠르게 파악 | 팀 출근 현황 + 승인 대기 |
| 원장 | 시설 최종 결정권자 | 밀린 결재를 빠르게 처리하고 전체 현황 파악 | 오늘 할 일 요약 + 오래된 대기 건 |
| HR관리자 | 인사·총무 담당 | 직원 등록, 규정 설정, 시스템 운영 전반 관리 | 시스템 설정 |
| 재단담당자 | 재단 본부 담당자 | 시설 전체 휴가 현황을 열람하고 보고서 다운로드 | 시설 현황 (열람 전용) |

## 5. 기술 사양

### 5.1 기술 스택

| 구분 | 기술 |
|------|------|
| 프론트엔드 | React + Noto Sans KR (내장) |
| 백엔드 | Node.js + Express |
| 데이터베이스 | SQLite |
| 패키징 | Node.js SEA (Single Executable Applications) → .exe |
| 통신 | REST API (JSON) |
| 폰트 | @fontsource/noto-sans-kr (풀 버전 내장, 인터넷 불필요) |

### 5.2 DB 테이블 (18개)

**인력 관리**
- departments — 부서/팀 (id, name, parent_id)
- employees — 직원 (id, employee_number, name, password_hash, role, department_id, hire_date, birth_date, gender[M/F], position, position_rank, phone, employment_type[regular/contract], status[active/inactive], is_absent, absent_return_date, is_initial_password, resignation_date, created_at, updated_at)

**휴가 유형**
- leave_types — 7가지 유형 (id, name, code, default_days, requires_attachment, allows_retroactive)
- condolence_subtypes — 경조사 세부유형 (id, leave_type_id, name, days, description)

**휴가 신청·결재**
- leave_requests — 신청 건 (id, employee_id, leave_type_id, condolence_subtype_id, start_date, end_date, half_day_type[null/AM/PM/TIME], time_start, time_end, total_days, reason, is_urgent, urgent_reason, is_retroactive, retroactive_category, retroactive_detail, status[draft/pending/approved/rejected/recalled/cancelled], parent_request_id, recall_reason, created_at)
- approval_flows — 결재 라인 템플릿 (id, name, description)
- approval_flow_steps — 라인 내 각 단계 (id, flow_id, step_order, step_type[draft/cooperation/review/approval], assignee_type[self/role/department/person], assignee_position, assignee_department_id, assignee_employee_id)
- approval_steps — 실제 결재 이력 (id, request_id, step_order, step_type, assigned_to, acted_by, is_delegated, status[pending/approved/rejected], approver_position, approver_dept_name, read_at, acted_at, comment)
- delegates — 대결자 지정 (id, employee_id, delegate_id, priority[1/2])

**잔여일수·조정**
- leave_balances — 연도별 잔여일수 (id, employee_id, year, total_days, used_days, adjusted_days, calc_detail)
- balance_adjustments — HR 수동 조정 이력 (id, balance_id, adjusted_by, amount, reason, created_at) ※ INSERT만 가능

**부가**
- notifications — 앱 내 알림 (id, employee_id, type, title, message, target_url, is_read, created_at)
- holidays — 공휴일 + 자체 휴일 (id, date, name, is_custom, year)
- system_settings — 관리자 설정 (id, key, value, description, updated_at)
- leave_type_flow_mapping — 유형↔결재라인 연결 (id, leave_type_id, approval_flow_id)
- export_logs — 내보내기 이력 (id, exported_by, export_type[usage/summary], date_from, date_to, verification_code, file_hash, created_at) ※ INSERT만 가능

**열람 범위 (v1.1 신규)**
- leave_request_visibility — 기안자 열람 범위 설정 (id, request_id, department_id). UNIQUE(request_id, department_id)

**고용형태별 휴가 설정 (v1.2 신규)**
- employment_type_leave_mapping — 고용형태별 사용 가능 휴가 유형 (id, employment_type, leave_type_id, is_allowed)

| BR-04-01 | 사후 신청은 아래 6가지 사유에 한하여 허용 (극히 이례적인 경우만) |
| BR-04-02 | 허용 사유 1: 본인 긴급 질병·부상 (복무규정 제28조) |
| BR-04-03 | 허용 사유 2: 전염병 격리 (복무규정 제28조①-2) |
| BR-04-04 | 허용 사유 3: 가족 사망·위독 (복무규정 제30조) |
| BR-04-05 | 허용 사유 4: 배우자 긴급 출산 (복무규정 제30조 + 별표) |
| BR-04-06 | 허용 사유 5: 천재지변·재해 (복무규정 제27조-5) |
| BR-04-07 | 허용 사유 6: 긴급 소집·법적 의무 (복무규정 제27조-3) |
| BR-04-08 | 사후 신청 시 사유 카테고리 6가지 중 선택 필수 + 상세 사유 텍스트 필수 |
| BR-04-09 | 증빙 서류 첨부 안내 (진단서, 사망신고서, 출생증명서 등) |
| BR-04-10 | 사후 신청 건은 결재 화면에서 "사후 신청" 배지 표시 |
| BR-04-11 | 연차(종일·반차·시간), 포상휴가, 생리휴가는 사후 신청 불가 |

### BR-05. 결재

| 규칙 | 내용 |
|------|------|
| BR-05-01 | 모든 단계가 순서대로 승인되어야 최종 승인 |
| BR-05-02 | 어느 단계에서든 반려 가능 → 전체 반려 |
| BR-05-03 | 반려 시 사유 입력 필수 |
| BR-05-04 | 취소는 최종 승인 전까지만 가능. 취소 시 건 종료, 재기안 불가 |
| BR-05-04a | 회수는 최종 승인 전까지만 가능. 회수 시 recalled 상태, 재기안 가능 |
| BR-05-05 | 최종 승인 후 취소 시 → 취소 재신청 절차 |
| BR-05-06 | 결재 이력은 수정·삭제 불가 |
| BR-05-07 | 반려된 건 → 재상신 가능 (이전 데이터 복사, 반려 사유 참고하여 수정 후 제출) |
| BR-05-08 | 회수된 건 → 재기안 가능 (이전 데이터 복사, 자유 수정 후 제출) |
| BR-05-09 | 재상신/재기안으로 생성된 건은 parent_request_id로 원본 연결. 이력 추적 가능 |

### BR-06. 대결 및 부재 관리

| 규칙 | 내용 |
|------|------|
| BR-06-01 | 결재자가 휴가 상태이면 대결자에게 자동 라우팅 |
| BR-06-02 | 결재자가 부재중 토글을 켠 상태이면 대결자에게 즉시 전환 |
| BR-06-03 | 1순위 대결자도 부재 시 2순위로 전환 |
| BR-06-04 | 대결자도 부재 시 원래 결재자에게 대기 (수동 처리 필요) |
| BR-06-05 | 대결 처리 이력에 원래 담당자 + 실제 처리자 모두 기록 |
| BR-06-06 | 미처리 자동 전환 시간: 긴급(휴가 시작 1일 이내) 즉시 / 단기(2~3일) 8시간 / 일반(4일+) 24시간 |
| BR-06-07 | 자동 전환 전 원 결재자에게 1차 미처리 경고 알림 (대기 시간의 50% 경과 시) |
| BR-06-08 | 부재중 토글의 복귀 예정일 경과 시 자동 해제 |
| BR-06-09 | 긴급 건은 부재 설정 여부와 무관하게 결재자 + 대결자 동시 알림 |

### BR-07. 열람 범위 (v1.1 신규)

| 규칙 | 내용 |
|------|------|
| BR-07-01 | 기안자가 신청 시 열람 범위를 팀별 체크박스로 설정 |
| BR-07-02 | 내 팀은 기본 선택 (해제 가능) |
| BR-07-03 | 승인된 건만 캘린더에 표시. pending/rejected/recalled/cancelled는 표시하지 않음 |
| BR-07-04 | 표시 정보: 이름 + 유형 + 기간 (사유는 비공개) |
| BR-07-05 | 열람 범위 미설정 시 어디에도 표시되지 않음 |
| BR-07-06 | 팀장·원장·HR은 역할 권한으로 전체 열람 가능 (열람 범위 설정과 별개) |

## 10. 제약 사항 및 가정

### 제약 사항
- 인터넷 연결 없이 동작해야 함
- 시설 PC(Windows) 환경에서 실행
- 외부 서비스(이메일, SMS, 클라우드) 연동 불가
- 재단 담당자에게는 엑셀 파일을 수동 전달 (1차)

### 가정
- 시설 내부 네트워크(LAN)가 구성되어 있음
- 시설 PC에 최소 1대에서 .exe 실행 (서버 역할)
- 브라우저는 Chrome 또는 Edge 최신 버전 사용
- 시설 당 직원 수는 10~50명 수준
- HR관리자가 최소 1명 지정되어 운영
